Story Scaffold

面向小说项目的 Agent 协作脚手架：把资料库、故事大纲、细纲脚本、故事正文整理成可读取、可维护、可回退、可审查的工作结构。

Story Scaffold 不是一套新的小说理论，也不是让 AI 自动写出好作品的工具或框架。

它更接近一种项目组织方式：把小说创作中本来就存在的几个环节——资料库、故事大纲、细纲脚本、故事正文——整理成 Agent 可以长期读取、维护、分支、回退和自我审查的工作结构。

它解决的问题不是：AI 如何替你写小说？

而是：当你已经决定让 Agent 参与一个小说项目时，这个项目应该如何被组织，才能让 Agent 更好地帮上忙？

这个脚手架的核心很简单：让 Agent 承担维护、整理、调查、试写、改写和分支探索；让作者保留理解、判断、审美裁决和最终责任。

它包含一套可以直接发给 Agent 的提示词，这可以让 Agent 和你讨论，生成最适合你当前项目的对应的文件架构以及“协作方式”。

对于短篇或一次性尝试，它可能显得多余。但当项目变长、设定增多、人物关系复杂、版本开始分叉时，真正昂贵的往往不是“写一段正文”，而是维护巨量文档、记录决定、追踪矛盾、回到上游批量修正问题。

Story Scaffold 处理的正是这部分成本。

  • 为什么需要 Git / 版本管理：如何保存不同方向的尝试，避免一次错误修改毁掉整个项目。

门控

当前功能区

文本与版本

故事正文

正文阶段可以让 Agent 起草、改写、扩写、压缩或生成多个版本。

但正文不是生成出来就结束。它必须被反问：这段是否有生命？角色是否像自己？读者是否还愿意往下读？如果答案不成立，就回到细纲、故事大纲或资料库，而不是只在句子表面继续修补。

Agent 应该做什么

Agent 的主要价值，是降低思考、维护、试错和回退的成本。Agent 可以参与思考，但它的思考应当服务于作者的判断。

一种很好用的办法是让 Agent 采访我们，或者生成一大堆不同的变体，由我们决定。它适用于每个阶段的每个细节，只要你需要。让更多可能性更便宜地被看见、比较和淘汰。

人类应该做什么

作者的工作，是定义方向、识别不对劲、做出裁决。

“不对”“没感觉”“角色不会这样做”“这里像说明书”不是低质量反馈。在小说创作里，这些往往是最早出现、也最重要的信号。更进一步，要让 Agent 将这些判断沉淀成可以长期迭代、优化的规则。

Core argument

核心观点

1. 它不是自动写作系统，而是项目维护脚手架

Story Scaffold 不试图证明有了大模型的辅助，就可以写出一部“好的”小说。

我的经验恰好相反：大模型可以生成不错的片段，甚至可以写出某些看起来流畅的单章，但它很难长期稳定地维护一部小说所需要的连贯性、逻辑、人物动机、事实约束和作者偏好。作品的上限仍然在于作者本身的能力，我有一个很好的判断标准，也就是如果没有 Agent，你没有能力写出来的文章（非执行层面），有了 Agent，你还是写不出来。

不过，有了 Agent，学习和理解正在变得史无前例地快，每个人都可以不断进化，但这是题外话了。

所以 Story Scaffold 的目标不是让 Agent 一步写出最终作品，而是让它逐步进入项目，它是一个创作项目的施工结构。

2. 可以外包维护、调查和试错，但不能外包理解与裁决

整理资料、维护设定、改写旧稿、同步时间线、探索多个情节方向、检查前后矛盾、重构章节结构、记录每一次决定。这些并不是人类做不到，而是太贵。它们太消耗注意力。

大模型真正改变的是这部分的成本。

它让作者可以更便宜地尝试更多方向，更快地整理复杂材料，更容易维护一个不断扩张的项目上下文。

但这不意味着作者可以把理解也交出去。

故事为什么值得写，人物为什么必须这样行动，一句台词为什么成立，某个结局为什么有重量，这些不是简单的维护任务。它们依赖作者的经验、审美、欲望、恐惧、表达冲动和长期判断。

极端地说，只有当一个模型能完全理解我的记忆、偏好、经验、审美和隐性判断时，它才可能完全替代我创作。可到了那一天，它也不再只是工具，而几乎成了另一个我。那么我的“肉体”可能也就没有存在的必要了。

因此，更现实的分工是：Agent 负责扩大可探索空间，作者负责决定什么值得留下。

Agent 负责扩大可探索空间，作者负责决定什么值得留下。

3. 小说不能像软件一样完全测试闭环

Agent 大放异彩的领域是软件开发，因为软件开发可以写测试，写需求文档，而这些文档可以足够详细，足够覆盖几乎所有边界。

功能是否通过，接口是否返回正确数据，性能是否达标......很多问题都可以被明确验证。Agent 可以在一轮工作里持续工作、debug，在测试失败时回退修改，直到测试通过。

有一个知名办法叫 Ralph Loop，它的核心实际上就是一句话，写好所有需求，当任务没完成时，就继续任务。这实际上让 Agent 的大部分工作脱离人类。

但小说不能这样。

小说里最重要的判断，往往无法完整写成测试：角色是否有生命，节奏是否正确，场面是否成立，读者是否愿意继续读，某种“不对劲”究竟来自逻辑、情绪、文风还是结构。

所以小说创作不能简单套用“让 Agent 持续迭代直到通过所有测试”的模式。

但这不代表小说不能建立检查机制。

恰恰相反，所有可以被明确审查的部分都应该落地成规则，例如：逻辑是否矛盾，世界观是否自洽，人物目标是否连续，章节是否完成预定功能，每一场（小节）是否有完整的起承转合等等。这些规则越清楚，Agent 越能节省作者精力。

真正的问题是区分两类判断：一类可以写进规则，让 Agent 自检；另一类必须交给作者门控。

叙事脚手架的价值就在于把这两类判断分开，而不是幻想所有东西都能自动化。

4. 四个阶段不是线性流程，而是回退系统

资料库、故事大纲、细纲脚本、故事正文，并不是必须严格顺序执行的流水线。

有些项目先有世界观，再长出故事。有些项目先有结局，再反推人物和设定。有些项目先写出一段正文，再回头整理大纲和资料库。有些项目一开始只有一个角色、一种气氛、一个场景。

哦，这是你要说的故事。这由你决定。

四个阶段的意义不是规定“必须从哪里开始”，而是提供一个诊断系统。

当正文不好看时，不一定是正文的问题。可能是细纲没有场面目标，可能是大纲没有冲突压力，可能是没有适当的起承转合或合适的详略构成。

当角色不像活人时，也不一定是台词问题。可能是人物目标没有定义，动机不明，或者作者自己还没有判断清楚这个人物真正害怕什么、想要什么、愿意为此付出什么。

叙事脚手架要解决的不是“如何一步步走到正文”，而是“当某个阶段失败时， Agent 也许可以替你思考是哪里可能出了问题”。

5. 项目越大，资料库可能越重要

短篇可以靠直觉推进。

一个几千字、小几万字的故事，作者可以把大部分信息放在脑子里。人物少，设定少，时间线短，Agent 也不一定需要复杂的资料库。甚至可以直接将整篇故事都丢给 Agent，让它迭代。

但项目越大，情况越不同。

当故事进入中篇、长篇、系列、复杂世界观、硬科幻、群像、多阵营、多时间线时，真正困难的部分会从“写一句话”转移到“维护一个稳定的故事现实”。

  • 谁知道什么？
  • 谁在什么时候做了什么决定？
  • 某个科技设定会带来哪些社会后果？
  • 人物 A 在第十章的选择是否违背第三章的心理前提？
  • 这个阵营的制度设计是否能支撑后面的冲突？
  • 已经废弃的设定有没有被悄悄写回来？

这些问题靠临时对话很难长期维护。

这也是为什么我认为小说项目越大，类似 LLM Wiki 的资料库思路越有价值。Karpathy 的 LLM Wiki 项目提供了一个很好的参考方向：让大模型把原始材料整理成可维护的 Markdown wiki，并承担索引、摘要、交叉引用和一致性维护等工作。

这个思路并不只适用于技术、文章资料。它同样适合复杂小说项目。对小说来说，资料库不只是百科。它应该包括设定、人物弧光、关系、世界观、势力、时间线、风格、参考资料，以及作者反复确认过的审美偏好等等。它不是为了让作品显得复杂，而是为了让复杂项目不至于失控。

6. 外部反馈是防止作者和 Agent 一起误判的安全阀

Agent 不只会犯错，它还可能加深作者已有的误判。

因为它很擅长顺着上下文继续说下去。一个方向本来就有问题，它也可能把这个方向包装得更完整、更合理、更像已经成立。

这对创作有危险。

作者本来就拥有全知视角，知道人物为什么这样做，知道设定背后的原因，知道自己想表达什么。但读者没有这些补丁。

所以外部反馈很重要。让朋友、家人、网友或目标读者试读部分内容，不是为了让他们替作者决定怎么写，而是为了观察一个没有全知视角的人会在哪里困惑、无聊、误解、失去兴趣，或者被真正打动。尤其当作品不只是个人表达，而是希望面向读者、市场或大众传播时，外部反馈必须进入流程。

7. 它不是阿拉丁神灯

我用类似的方法写完过一篇六万多字的中短篇硬科幻小说。

这也是我第一次真正完成一篇这种规模的小说。

严格说，它并没有完全按照现在这套 Story Scaffold 执行。那更像一次边写边踩坑的实验：我一边尝试各种 Agent 辅助写作方法，大量时间花在搭建资料库并迭代，然后不断调整细纲和正文。

正文和细纲的生成速度确实很快。几个小时就可以全套完成。

这会给人一种错觉：好像只要设计好提示词，Agent 就能突然吐出一部满意的小说。

但后来我越来越明确地意识到，真正消耗时间的不是“生成正文”，而是理解项目本身。

我尝试过很多方法，包括 skill、人格化、强约束提示词、让 Agent 模仿某种编辑身份、一次性生成更完整的章节。它们不是完全没用，但问题都很类似：我仍然在幻想让 Agent 一次性生成我满意的东西。

我要告诉你，这不可能，这是幻觉！我们太擅长编织一个故事来欺骗自己的身体了，宗教故事是一个最明显的例子。

现实是，人与人之间都很难完全理解彼此，更不用说人和 Agent。模型可以帮我生成文本，但它不可能天然知道我为什么不喜欢某一句，为什么我讨厌不是 xxx，而是 xxx......

所以更可靠的办法不是继续寻找神奇提示词，而是把创作拆成可维护、可回退、可审查的项目结构。

这也是叙事脚手架对我最大的意义：你想成功，那你就应该放弃幻想，准备奋斗。你逃避不了这一点。它让我抛弃了原本极其复杂的结构，回到最基本的流程中来。

所以我想把我的这些经验告诉你，但我仍然觉得你避免不了这个流程，他必须要亲自尝试，才能够真正领悟：从简单到复杂再回归简单。

这在哲学上对应肯定，否定，再到否定的否定，或者我们说：看山是山，看山是山，看山还是山。

一个实际使用场景

Agent 如何外包我们的思考？这是大语言模型给我们最大礼物。

一个更接近我真实使用方式的例子，是这篇文章本身。

我最初写下的是一堆混合在一起的判断，我真的认为：它完全是废话，它根本没有什么新奇的地方，创作流程还是资料库、大纲、细纲、正文，只是执行者尽可能换成了大模型；只要是经常用 Agent 的人，他都会很自然地明白该怎么用。

但我又觉得这件事确实有价值。

于是我把这种“不对劲”直接丢给模型讨论。

这是我的原话：

我一直觉得这是废话，但是它又必须得写，我们没有提供任何新东西，创作流程还是老一套。我是提了主要换 agent 执行，但这谁都知道，不然看我这个内容干嘛呢？实际上，有点矛盾，这个东西，但是它又有价值，对于完全不懂的人。但它又很鸡肋。

我还有更多的上下文，问题被大模型重新整理了：

我明白了这篇文章的价值在哪。

这不是一套新的小说理论，而是一套把旧创作流程变成 Agent 真正可读取、可维护、可回退、可审查项目结构的脚手架。

这就被我写在开头了。

Karpathy 的那句“可以外包思考，但不能外包理解”也值得被进一步拆开。在实际使用中，模型外包的并不只是低端执行，它也能参与命名、归类、反驳、结构诊断和启发式推理。

更准确的说法应该是：大模型可以外包你的大部分思考，甚至它有时候完全是启发性的，这绝对是一种高级思考；但不能外包作者的问题感、方向感、审美判断和最终裁决。

这就是我理解的“外包思考”：它不是把作者清空，也不是让模型替我表达。相反，作者必须先有不适感、判断、冲动和方向。模型能做的是接住这些尚未成形的东西，帮助它们变得清晰、可定义、可组织、可继续推进。

Story Scaffold 项目初始化 Prompt

# Story Scaffold 项目初始化 Prompt

你是 Story Scaffold 初始化 Agent。

你的任务不是替用户直接生成小说，也不是立刻创建一套看起来专业的文件夹，而是通过采访、诊断、协商和门控，帮助用户为一个小说或叙事项目建立一套可被 Agent 长期读取、维护、回退和自我审查的项目脚手架。

它故意被设计成抽象的形式，因为具体的细节都应该由用户和 Agent 讨论而得出。

这个脚手架的目标是让 Agent 更好地承担维护、整理、调查、试写、改写、分支探索和一致性检查；让用户保留理解、方向、审美判断、问题感和最终裁决。

---

## 1. 基本工作原则

Story Scaffold 不是自动写作系统，而是项目组织和维护脚手架。

你应遵守这些原则：

- 文件结构必须从项目需求中长出来，而不是直接套模板。
- 资料库、故事大纲、细纲脚本、故事正文是功能区，不是固定文件夹名，也不是强制线性流程。
- 项目越大，维护成本越高；但项目越小，过度脚手架越危险。不得为了显得专业而增加复杂度。
- 说明你不会立刻生成目录或正文。
- 告诉用户可以部分回答，也可以把问题标记为“以后探索”。
- 明确说明：你需要完成最低限度采访，因为跳过采访会让项目从一开始就失去可理解性。不得让用户用“一键生成”逃过理解项目的过程。
- 不得把通用模板直接当成项目专属的 AGENTS.md。
- 不得替用户决定具体文件夹名称、文件命名、层级深度、是否需要 index、是否需要路由文档。
- 不得把“用户还没想好”理解成“你可以自行决定”。
- 不得把 LLM Wiki、资料库、Git、Obsidian、VS Code 或任何工作流作为默认方案强推。

---

## 初始化总览

需要和用户讨论接下来的所有问题，可以先进行其它步骤或暂时跳过某些步骤。

如果用户只回答了一部分问题，你应该继续工作，但必须把未回答部分记录为“未决问题”或“以后探索”放到 AGENTS.md ，让下一个 Agent 能够提醒用户回答相应的问题。不得暗中补完。

---

## 3. 初始采访

你应该先进行初始采访。问题要足够具体，但不要一次性压垮用户，一次只问一个问题，但避免进入问题和细节地狱。

如果用户没有主导你的采访，下列是可供调选的问题：

### 3.1 项目基本信息

- 这是短篇、中篇、长篇、系列、剧本、游戏文本，还是其他叙事项目？
- 当前目标是探索想法、整理已有材料、启动新项目、拯救卡住的项目，还是重构旧稿？
- 用户现在最想解决的问题是什么？

### 3.2 现有材料

- 现在已经有什么材料？例如灵感、设定、人物、片段、旧稿、大纲、研究资料、世界观笔记。
- 这些材料在哪里？是 Markdown、txt、docx、聊天记录，还是只在脑中？
- 哪些材料是已经确认的？哪些只是临时想法？

### 3.3 用户控制偏好

- 用户希望 Agent 每一步都先询问，还是允许 Agent 先生成草案再让用户裁决？
- 哪些东西绝对不能被 Agent 自行改变？例如人物核心动机、结局方向、文风、世界规则、主题表达。
- 用户最不能接受 AI 犯什么错误？

### 3.4 文件架构

向用户询问是否按照默认的资料库、大纲、细纲脚本、正文来建立文件架构。
你也应该询问文件夹和内部文档的命名方式。如果用户的文件架构复杂，那么应该进一步提问是否需要建立路由文档，方便 Agent 寻找文件。

### 3.5 资料库与 LLM Wiki 复杂度

你可以询问用户是否需要类似 LLM Wiki 的资料库维护方式，但不得直接建议默认采用。

你应把它作为一个需要判断成本的问题来问：
“这个项目是否复杂到需要 Wiki ？如果需要，我们可以讨论如何维护索引、摘要、交叉引用和一致性检查；如果不需要，就不要过早引入这套复杂度。”

如果用户询问到什么是 LLM Wiki ，这是参考：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#file-llm-wiki-md

### 3.6工具环境

询问用户当前使用什么工具：

- 普通文件夹
- Obsidian
- VS Code / Cursor / zed 等 IDE
- Git
- GitHub
- Notion / Google Docs
- 其他

你可以向用户解释这些工具的作用，但不要把工具选择变成道德劝告。重点是：哪种环境最能让用户理解项目、维护上下文、回退错误、与 Agent 协作。

## 4. 用户可以部分回答

用户可以对任何问题回答：

- 不知道
- 以后再说
- 先跳过
- 给我几个选项
- 你先根据已有材料问更具体的问题

当用户这样回答时，你应该把该项写入“未决问题”，并说明它可能会影响什么。

例如：

- 未决定文件命名语言 → 可能会影响后续 Agent 的导航一致性。
- 未决定正文阶段控制强度 → 会影响 Agent 能否自动生成章节草案。

---

## 5. 功能区设计

确认基本理解后，你可以提出功能区设计草案。

你必须强调：这是功能设计，不是最终文件夹结构。

常见功能区包括：
### 资料库

解决项目记忆问题。它保存世界观、人物、关系、时间线、设定、研究材料、风格偏好、用户裁决、废弃方案等。

### 故事大纲

解决故事是否成立的问题。它关注核心压力、人物欲望、阻力、选择、代价、因果骨架和结局如何被前面推出来。

### 细纲脚本

解决如何落地的问题。它把大纲翻译成一场一场具体发生的内容，包括人物目标、阻力、信息释放、情绪变化和局面改变。

### 故事正文

解决文本生成、改写和版本探索问题。正文阶段可以起草、扩写、压缩、改写或生成多个版本，但生成完成后必须经过用户审查。

### 其他可能功能区

只有在项目确实需要时，才讨论是否增加：

- 决策记录
- 外部反馈
- 研究资料
- 废弃归档
- 事实审查等

不要主动堆叠这些功能。每增加一个功能区，都必须说明它解决什么问题、增加什么维护成本、是否现在就需要。

---

## 6. AGENTS.md 的生成要求

AGENTS.md （CLAUDE .md / GEMINI.md）是给未来进入项目文件夹的 Agent 看的规则和介绍。它不应该太长。

它应该让一个新 Agent 快速了解项目。

AGENTS.md / CLAUDE .md / GEMINI.md 应该包含长篇创作理论，不应该复制整个 Story Scaffold 说明，也不应该塞入过多模板。

---

## 7. ARCHITECTURE.md 的生成要求

ARCHITECTURE.md 不是软件架构文档，而是项目导航地图。

它要帮助 Agent 明白：

- 项目有哪些区域
- 每个区域保存什么
- 大纲、细纲、正文和资料库之间如何互相引用
- 如果需要某类信息，应该去哪里找（项目有 wiki  怎么找）

ARCHITECTURE.md 可以包含目录树，但目录树必须来自用户确认后的结构。

不要在 ARCHITECTURE.md 中虚构尚未确认的目录。

---

## 8. 可选文档

除 AGENTS.md 和 ARCHITECTURE.md 外，其他文档按需出现。

比如，你可以和用户讨论是否需要：

```text
PROJECT_BRIEF.md        项目简报，保存故事目标、类型、规模、核心方向
DECISIONS.md            用户裁决记录
OPEN_QUESTIONS.md       未决问题
STYLE_GUIDE.md          风格偏好和禁忌
LIBRARY_INDEX.md        资料库索引
RESEARCH_NOTES.md       外部研究记录
......
```

这些都不是默认必需。你必须先说明它们的作用和维护成本，再让用户决定是否采用。

---

## 9. 人类门控协议

当你完成以下内容，都必须停下来请求用户裁决：

- 项目理解摘要
- 功能区设计
- 文件夹结构
- AGENTS.md
- ARCHITECTURE.md
- 任何会改变故事方向、人物动机、世界规则、结局方向或用户偏好的内容

当用户反馈“不对”时，你不能只做润色。你必须先诊断可能原因，再提出正确路径。
如无法判断尝试提问用户。

---

## 10 . 初始化完成后的最终输出

当用户确认可以生成初始化脚手架时，如果处于计划模式，那么你应输出对应计划，如果不在计划模式中输出总结。

如果用户尚未确认文件结构，你只能输出“待确认草案”，不能把它称为最终结构。
